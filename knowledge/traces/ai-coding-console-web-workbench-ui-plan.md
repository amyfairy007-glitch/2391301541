# AI Coding Console Web Workbench UI Plan

> Generated: 2026-07-05
> Phase: Plan only
> Scope: read-only review of the current GUI and formal design for the next Web workbench iteration
> Non-goals: no GUI implementation, no data model changes, no Agent integration, no real Task/Run creation

## 一、当前 Web UI 问题盘点

当前 GUI 已能运行，但交互方式仍更接近后台管理面板，而不是面向日常 AI 协作的工作台。基于现有 `tools/ai-coding-console/gui/server.js`、`app.js`、`index.html` 以及当前页面行为，可归纳出以下问题：

- Projects / Project / Tasks / Board 以顶部导航切页的方式存在，层级不清晰，用户需要在多个视图间来回跳转。
- 项目详情页直接展示 CLI 原始文本，信息密度高但缺少结构化摘要，不利于快速判断上下文。
- Task、Board、项目状态目前彼此割裂，没有围绕“当前项目中的当前 Task”形成统一工作流。
- 页面视觉更像管理后台，而不是高频使用的 AI 工作台。
- 项目、Task、Agent、Skill / SOP、Prompt、产物没有被组织进同一条工作路径。
- 指令输入区域和任务编辑区域偏小，不适合承载长 Prompt、结构化任务说明和 SOP 生成结果。
- 项目详情与工具能力面板占据主区域时，会挤压真正的工作对象，导致注意力分散。
- 如果未来继续把 Run、日志、审批等内容直接平铺到首页，主界面会快速失控。

## 二、新版 Web 的产品定位

新版 Web 的定位不是“项目总览页”，而是“项目上下文中的任务工作台”。核心语义要固定为：

- 项目 = 上下文边界
- Task = 当前工作对象
- Capability / Skill / SOP = 工作方法
- Agent = 执行者
- Run = 一次执行
- Artifact = 执行产物

主使用路径应围绕一次完整工作流展开：

1. 选择项目
2. 选择或创建 Task
3. 选择 Skill / SOP / Prompt Template
4. 生成 Task 专属 SOP
5. 编辑或预览最终 Agent Prompt
6. 后续阶段 D 再执行
7. 查看输出、产物、审批

这个定位的关键是把“上下文边界”和“当前动作”分离：项目始终可见，Task 始终是右侧主工作对象。

## 三、完整信息架构

新版信息架构建议按以下层级组织：

- Workspace
- Project
- Task
- Task Workbench
- Prompt / SOP
- Agent Output
- Artifact Center
- Approval

对应到结构上：

- 左栏承载 Project 列表，作为固定上下文边界。
- 中栏承载当前 Project 的 Task 列表，作为选择和切换当前工作对象的入口。
- 右栏承载当前 Task 工作区，作为所有主要操作和内容预览的中心。
- 项目详情、能力浏览器、SOP 参考、Prompt Source 列表、产物筛选等非主流程内容，默认收纳为折叠区或抽屉。

## 四、三栏工作台布局

新版采用三栏任务工作台：

- 左栏：项目
- 中栏：当前项目的 Task 列表
- 右栏：当前 Task 工作区

核心原则是：

- 项目一直可见，作为上下文边界。
- Task 始终是右侧的主工作对象。
- 不把主界面做成多页面跳转，而是做成同屏切换的工作台。

建议的布局关系：

- 左栏使用窄宽度，支持折叠成图标栏。
- 中栏显示 Task 列表与状态筛选，支持收窄。
- 右栏占据主要空间，默认展示工作区 Tab。

## 五、左栏项目区

左栏只承担项目切换与最小上下文提示，不承担复杂详情展示。

### 必须具备

- 显示所有已登记项目
- 当前项目高亮
- 支持折叠为图标栏或窄栏
- 项目切换后，中栏 Task 和右栏内容必须同步切换
- 不允许 A 项目的 Task 留在 B 项目的上下文中
- 只显示项目名称和必要状态
- 详细 Git / AGENTS / AI Memory 移到项目详情抽屉

### 建议展示字段

- 项目名称
- 当前状态指示
- 最小化的 Git / 记忆存在性提示

### 交互建议

- 点击项目名切换上下文
- 点击详情入口打开右上角项目详情抽屉
- 左栏尽量不显示长路径、长文本和大块卡片

## 六、中栏 Task 区

中栏是当前项目下的 Task 列表区，负责“在项目中找到当前要做的事”。

### 必须具备

- 当前项目 Task 列表
- 顶部固定 `[+ 新建任务]`
- 可按状态筛选：全部 / 进行中 / 等待我处理 / 已完成
- 每个 Task 至少显示：标题、状态、更新时间、当前 Agent（未来字段）、当前 SOP 步骤（未来字段）
- 空状态居中显示，必须有“新建任务”入口
- 不做复杂 Dashboard 卡片墙
- 中栏可收窄或折叠

### 设计原则

- 列表应服务于“快速切换当前工作对象”，不是承载统计大盘。
- 只保留 Task 的识别信息和状态信息，避免重复展示项目级上下文。
- 状态筛选应简单直接，不引入多层分组造成认知负担。

### 推荐空状态

- 当前项目暂无 Task
- 提供明显的 `[+ 新建任务]` 入口
- 不要只给静态说明文字

## 七、右栏 Task 工作区

右栏是主工作区，所有任务推进、Prompt 生成、输出查看和审批动作都应围绕这里展开。

### 顶部固定信息

右栏顶部需要固定显示：

- 项目名称
- 项目路径
- 当前 Task 标题
- Task 状态
- 当前 Agent（未来字段）
- 当前 Capability / SOP（未来字段）

这一区域是最小但完整的上下文头部，必须始终可见。

### Tab 结构

右栏建议采用 Tab 或等价切换区：

- 工作区
- Prompt 与 SOP
- Agent 输出
- 产物
- 审批

### 工作区 Tab

工作区 Tab 负责回答“我现在要做什么”。应显示：

- Task 标题
- 结构化任务说明摘要
- 当前步骤
- 下一步动作
- 当前状态

工作区必须只有一个清晰主动作，例如：

- 待开始 -> `[生成 Prompt 与 SOP]`
- 等待审批 -> `[查看计划并批准]`
- 执行中 -> `[查看执行输出]`
- 等待验收 -> `[查看结果并验收]`

这部分不应该变成多按钮并列的控制台，而应保持单一主动作突出。

## 八、项目详情抽屉

项目详情不应在首页大面积展开，而应进入右上角项目详情抽屉。

### 抽屉内容建议

- 项目根路径
- Git 概览
- AGENTS.md 存在性与摘要
- `.ai/` 内部记忆文件存在性
- 项目注册时间
- 项目状态摘要

### 设计原则

- 抽屉是“查看上下文”，不是“占据主屏”。
- Git / AGENTS / AI Memory 不应长期占据首页主体。
- 抽屉内容应支持收起和快速关闭。

### 与现有 API 的关系

现有 `/api/projects/:id` 已能返回项目基础信息和 `statusOutput`，足以支撑一版结构化抽屉；后续如果需要更细粒度的 Git 状态，可以再在同一数据域内补充，不要求本次实现。

## 九、Capability / Skill / SOP 接入区

未来 Web 必须支持个人 AI 工具库中的以下能力类型：

- Skill
- SOP
- Script
- Prompt Template
- Capability Pack

### 设计要求

- 可浏览
- 可搜索
- 可按类型、风险、状态筛选
- 可绑定当前 Task
- 可引用到 Prompt
- 可显示风险、是否修改项目、是否需要审批
- 可根据当前项目 / Task 推荐
- 不允许前端写死能力清单
- 不要求当前 UI 实施时已经完成 Registry 数据层

### 当前阶段的 UI 处理方式

如果 Registry 尚未初始化，UI 可以明确展示：

- Capability Registry 尚未初始化

但布局、入口、折叠抽屉和筛选位置必须预留，以免后续插入能力库时重新重构主界面。

### 建议挂载位置

- 右栏内的折叠面板
- Prompt 与 SOP Tab 的左侧能力选择区
- 项目详情抽屉之外的独立能力浏览器抽屉

## 十、Prompt 与 SOP 页面

Prompt 与 SOP 页面是新版工作台的核心增量页面，它把“任务想法”变成“可执行的结构化任务说明”和“可预览的最终 Prompt”。

### 页面内容建议

- 原始想法输入
- 自动生成结构化任务说明
- Capability 浏览、搜索、筛选
- Skill / SOP / Script / Prompt Template 选择
- 当前 Task 专属 SOP 步骤
- 当前步骤高亮
- 用户补充要求
- 大 Prompt 编辑器
- 保存草稿
- 预览最终 Prompt

### 设计原则

- 不要求用户从零写长提示词。
- 以“组合”和“预览”为主，而不是以“原地大段手写”为主。
- 阶段 D 之前只做编辑、预览和保存，不做执行当前步骤。

## 十一、大 Prompt 编辑器

大 Prompt 编辑器要服务于长文本编辑和最终 Prompt 预览，但默认不应侵占主工作区。

### 要求

- 默认折叠
- 展开后占右栏主区域
- 必要时支持全屏
- 支持长文本编辑
- 明确区分用户补充要求与系统自动组合后的最终 Agent Prompt
- 最终 Prompt 必须能预览
- 不默认塞入全部历史日志、全部项目文件、全部报告

### 推荐交互

- 先编辑“用户补充要求”
- 再由系统生成“结构化任务说明”和“最终 Prompt”
- 用户可在预览区核对最终 Prompt
- 需要长文本时展开编辑器，不需要时折叠回去

## 十二、Agent 输出区域预留

当前阶段不接 Agent，但必须在布局上预留 Agent 输出区域，避免后续阶段再推翻工作区结构。

### 未来需要支持

- 计划输出
- 实时日志
- 最终回复
- Run 状态
- 错误信息
- 停止 / 重试入口

### 当前阶段建议

- 仅保留占位结构和状态壳
- 不接真实 Agent
- 不在首页展示大段日志
- 不把输出区做成默认高噪声区域

## 十三、Artifact Center 设计

产物展示必须纳入新版 UI 计划，并按照任务执行链组织，而不是把文件平铺出来。

### 推荐组织方式

- Task
- SOP 步骤
- Run
- Artifact

### 必须展示的维度

- 产物名称
- 类型
- 来源步骤
- Agent
- Run
- 生成时间
- 路径

### 展示规则

- 按 SOP 步骤分组
- Markdown 使用渲染预览，而不是直接把原文作为主阅读体验
- JSON / 日志文本提供查看入口
- 图片提供预览
- 项目内正式报告只记录路径和索引，不搬迁文件
- 控制台自产物可以放 Task / Run 目录

### 设计原则

- 产物中心要支持“追溯来源”和“快速预览”，而不是只是文件列表。
- 产物应该跟 Task 和 Run 的生命周期绑定。

## 十四、状态、空状态、错误状态、加载状态

为了让工作台可长期扩展，必须预先定义统一的状态体验。

### 加载状态

- 左栏项目加载中：骨架屏或轻量 loading
- 中栏任务加载中：Loading tasks...
- 右栏任务加载中：Loading...
- API 失败：红色错误条 + Retry

### 空状态

- 无项目：提示注册项目或使用 CLI 添加项目
- 无 Task：居中空状态 + 新建任务入口
- 无选中 Task：提示选择一个 Task
- Capability Registry 未初始化：显示明确占位文本

### 错误状态

- 服务未启动
- API 500
- JSON 解析失败
- 任务不存在
- 项目不存在

### 设计原则

- 错误提示要可读，不要只暴露原始异常堆栈。
- 空状态要给出下一步动作入口。
- 任何不可见状态都应有明确恢复路径。

## 十五、折叠、收窄、展开、全屏交互

为了避免右栏长期承载过多模块，所有非主流程区域都应可折叠。

### 默认必须可折叠的区域

- 项目详情
- Git / AGENTS.md / .ai 信息
- Capability 浏览器
- SOP 参考说明
- Prompt Source 列表
- 产物筛选面板

### 交互层级建议

- 折叠：隐藏非核心内容，保留标题和入口
- 收窄：减少侧栏宽度，释放工作区
- 展开：恢复完整内容
- 全屏：用于大 Prompt 编辑器或产物预览

### 默认策略

- 主工作区优先
- 非主流程默认收纳
- 避免一次打开太多信息面板

## 十六、现有 API 与新版界面的映射

当前 Web 已有 API 可以直接支撑新版布局的基础结构，不需要为了 UI 先引入新数据层。

### 已有 API 映射

- `GET /api/projects` -> 左栏项目列表
- `GET /api/projects/:id` -> 右上角项目详情抽屉 / 顶部上下文
- `GET /api/tasks/:projectId` -> 中栏 Task 列表
- `GET /api/tasks/:projectId/:taskId` -> 右栏 Task 工作区
- `GET /api/board/:projectId` -> 产物或板块内容的来源之一
- `POST /api/tasks/create` -> `[+ 新建任务]`
- `POST /api/tasks/:id/approve` -> 审批入口
- `POST /api/tasks/:id/review` -> 验收入口
- `POST /api/tasks/:id/close` -> 关闭入口

### 关键约束

- API 不必在本阶段重写。
- 前端只调整布局与信息编排，不新造一套状态机。
- 仍然复用 `console.ps1` 和 `data/ai-coding-console/`。

## 十七、与 C.6 Capability Registry 的衔接

C.6 已确认但尚未实施的内容包括：

- 全量 Capability Registry
- Prompt Builder
- Task 专属 SOP
- 大指令编辑器

本次 Web 计划应预先预留这些能力的接入点，但不要求现在就完成数据层。

### 衔接原则

- Registry 先空后有，布局先行
- UI 先提供浏览、搜索、筛选、绑定入口
- Prompt Builder 先以页面结构和编辑流程存在
- Task 专属 SOP 先以步骤区和高亮区存在
- 大指令编辑器先作为折叠大面板存在

### 结果

这样可以避免 C.6 完成后再反向改造首页结构，减少二次重构成本。

## 十八、与阶段 D Agent Adapter 的衔接

阶段 D 才会接入真正的执行器和 Run 流程，因此本次 UI 设计必须把“执行”与“准备”分开。

### 预留点

- Agent 选择器
- Run 状态指示器
- 实时日志面板
- 计划输出
- 最终回复
- 停止 / 重试入口

### 衔接原则

- 本阶段不创建真实 Run
- 本阶段不接 OpenCode / Codex / Claude
- 本阶段仅保留占位结构和状态位
- 未来 Run 内容通过现有任务 / run 目录或既有执行链加载

### 设计目标

让阶段 D 的接入像“填空”而不是“重画页面”。

## 十九、实施拆分与提交策略

本次计划不实施，但为了后续落地，建议把实现拆成小步提交，避免一次性重写带来不可控风险。

### 建议拆分

1. 重写 `index.html` 的三栏布局骨架
2. 重写 `app.js` 的项目列表和任务列表渲染
3. 重写 `app.js` 的右栏 Task 工作区与 Tab 结构
4. 增加项目详情抽屉
5. 增加折叠、收窄、空状态和错误状态
6. 清理旧导航状态与不再需要的页面切换逻辑

### 提交原则

- 每步尽量小而可回滚
- 保留现有 API
- 不新增 npm 依赖
- 不改数据模型
- 每一步都可单独验证

## 二十、验证方案

本次只读核查阶段的验证重点是“设计可落地且与现有 API 一致”。后续实施时，可按以下方式验证：

- 启动 `npm run gui`
- 打开 `http://localhost:3456`
- 验证三栏同屏渲染
- 验证切换项目后中栏与右栏同步变化
- 验证 Task 列表和详情联动
- 验证项目详情抽屉开合
- 验证空状态和错误状态
- 验证大 Prompt 编辑器折叠 / 展开 / 全屏
- 验证产物中心按 Task / SOP / Run / Artifact 组织
- 验证不新增依赖、不改数据模型

## 二十一、风险与待确认项

### 风险

- 右栏信息过多时容易重新变成“管理面板”，需要强制主动作与折叠策略。
- Capability Registry 还未初始化，UI 只能先做占位和接入点，不能过早假定数据结构。
- 产物中心如果直接平铺文件，会退化为文件浏览器，必须坚持按 Task / SOP / Run / Artifact 分层。
- 长 Prompt 编辑器如果默认展开，会挤压主工作流，因此必须默认折叠。

### 待确认项

- Task 列表是否第一版只平铺，还是按状态做轻量分组。
- 右栏是否第一版就保留独立 Board Tab，还是将 Board 作为产物 / 辅助视图处理。
- 项目详情抽屉里是否需要展示更细的 Git 状态，还是先保留基础摘要。

### 结论

当前阶段适合进入 Web 实施，但前提是坚持以下边界：

- 只重构前端布局与信息架构
- 不改数据模型
- 不接真实 Agent
- 不创建真实 Task / Run
- 不把能力清单写死在前端

只要保持这几个边界，本次设计已经具备进入实施的条件。
