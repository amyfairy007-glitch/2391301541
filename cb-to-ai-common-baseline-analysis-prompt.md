# CB 对齐结果 → AI 库公共实现基线：专项分析提示词

请执行一次“CB 对齐结果 → AI 库公共实现基线”的专项分析。

## 一、当前背景

当前已经完成：

1. 成熟库自身分析；
2. AI 库自身分析；
3. 成熟库与 AI 库的对比分析；
4. 整个 CB 模块与成熟库的初步对齐。

本轮不要重新执行以上分析，也不要继续对 CB 做大范围修改。

当前目标是：

基于 CB 已经完成的初步对齐结果，识别其中哪些实现属于 CB 业务特例，哪些属于多个 journey 都可以复用的公共模式；再与成熟库中的公共实现方式核对，形成 AI 库后续页面治理和迁移可统一遵循的公共基线。

## 二、本轮需要回答的问题

请重点回答：

1. CB 对齐过程中使用了哪些公共能力？
2. 这些能力在成熟库中是如何组织和使用的？
3. CB 当前做法是否已经符合成熟库的公共模式？
4. 哪些做法可以上升为所有 journey 的统一基线？
5. 哪些做法只是 CB 的业务特例，不能推广？
6. AI 库当前缺少哪些公共封装、统一规则或标准样例？
7. 下一步应该用哪个非 CB journey 验证这套公共基线？

## 三、硬性限制

必须遵守：

1. 本轮只做分析，不修改业务代码。
2. 不重新分析整个成熟库。
3. 不重新分析整个 AI 库。
4. 不重新执行成熟库与 AI 库的全量对比。
5. 不继续做 CB 的业务功能开发。
6. 不批量修改其他 journey。
7. 不修改项目顶层结构。
8. 不引入新依赖。
9. 不复制成熟库业务代码到 AI 库。
10. 不默认认为成熟库所有写法都必须照搬。
11. 不根据命名或印象猜测公共模式，必须提供真实代码路径和调用证据。
12. 如果现有分析报告中已经有结论，优先复用，不重复扫描。
13. 如果无法区分公共能力和业务特例，标记为“需要人工确认”。

## 四、读取顺序

请严格按以下顺序读取。

### P0：已有总结和分析结果

优先定位并读取：

- 当前项目 Guide；
- CB 初步对齐报告；
- AI 库分析报告；
- 成熟库分析报告；
- 成熟库与 AI 库对比报告；
- 已有组件映射、API、validation、buildRequest、页面结构相关报告；
- CB 本轮修改或验收报告。

先从已有结论中确定：

- CB 已经改了什么；
- 哪些差距已经处理；
- 哪些问题仍未解决；
- 哪些实现被认为是公共模式。

### P1：AI 库中的 CB 实际代码

只读取 CB 当前实际涉及的代码，识别其使用的公共能力，包括但不限于：

- 页面容器；
- 路由或模块注册；
- 页面初始化；
- Form / FormItem；
- Input / Select / DatePicker；
- Table / columns；
- Modal / Drawer；
- Button；
- Pagination；
- validation；
- loading / disabled / readonly；
- Message / Notification / 错误提示；
- API 调用；
- buildRequest；
- response adapter；
- constants；
- strings；
- utils；
- dev / sit；
- stub / mock；
- 页面布局；
- 公共组件引入方式。

不要无差别扫描所有 journey。

### P2：成熟库中的对应公共实现

针对 P1 中实际识别到的公共能力，只在成熟库中查找对应实现：

- 相同公共组件的真实用法；
- 相同页面结构；
- 相同表单场景；
- 相同表格场景；
- 相同弹窗场景；
- 相同 API 和请求构建模式；
- 相同 validation 模式；
- 相同环境与 stub 切换方式；
- 相同文件职责划分；
- 多个 journey 都在复用的公共封装。

不要读取与 CB 和后续 journey 公共能力无关的成熟库业务模块。

## 五、分析范围

### 1. 页面公共骨架

检查：

- journey 的入口如何组织；
- 页面容器如何使用；
- 页面初始化在哪里执行；
- 查询区、表单、表格和操作区如何组合；
- loading、空状态和错误状态如何放置；
- CB 当前页面骨架与成熟库是否一致；
- 哪些结构可作为所有 journey 的基础骨架。

### 2. 公共组件使用方式

针对 CB 实际使用的公共组件，检查：

- import 来源；
- props；
- 数据格式；
- 默认值；
- disabled / readonly；
- validation；
- 事件回调；
- 表单绑定；
- columns 定义；
- modal 状态；
- pagination；
- API 数据适配；
- 是否存在两套组件库混用；
- 是否存在页面内临时封装。

需要区分：

- 成熟库稳定公共用法；
- CB 当前用法；
- 可推广的统一规则；
- CB 业务特例。

### 3. API 与数据流

检查：

- 页面状态如何转换为请求参数；
- buildRequest 是否承担统一职责；
- API 调用是否集中；
- response 是否需要适配；
- 页面是否直接依赖接口原始结构；
- 空值、日期、金额、布尔值如何处理；
- 错误码和异常如何处理；
- 成熟库是否有多个 journey 共用的封装。

### 4. validation 与交互状态

检查：

- validation 定义位置；
- 触发时机；
- 错误展示；
- 按钮禁用条件；
- 重复提交保护；
- loading 状态；
- modal 关闭后的状态清理；
- 查询条件变化后的旧数据清理；
- 成熟库是否存在统一模式。

### 5. 文件职责

检查并比较：

- `index.jsx`
- `buildrequest.js`
- `constants.js`
- `strings.js`
- `utils.js`
- API 文件
- stub 文件
- validation 文件
- columns / config 文件（如果项目实际存在）

判断：

- CB 当前拆分是否适合推广；
- 成熟库实际职责边界；
- 哪些内容应成为 AI 库最低统一规则；
- 哪些拆分只适用于复杂页面，不应强制所有 journey 使用。

### 6. 环境与测试数据

检查：

- dev / sit 边界；
- stub / mock 的实际使用；
- 数据切换方式；
- API 和本地数据是否混用；
- 成熟库中对应方式；
- 是否存在可以统一的环境基线。

## 六、公共模式判断标准

只有同时满足以下条件之一，才能建议上升为公共基线：

1. 成熟库中多个 journey 使用相同模式；
2. 成熟库已有明确公共封装；
3. 组件 README、类型声明和成熟库真实代码一致；
4. CB 当前实现已经验证可运行，并与成熟库同类场景一致；
5. 现有分析报告已明确确认该模式。

以下情况不得直接上升为公共基线：

- 只在 CB 中出现；
- 只在成熟库单个业务页面出现；
- 与特定业务字段或流程强绑定；
- 依赖成熟库特有版本或历史封装；
- 没有运行或代码证据；
- 不同成熟库页面存在明显不同做法；
- 当前 AI 库依赖版本不支持。
