## 1. 项目定位

`ai-ui-agentic` 是一个个人全局公用的 AI 工程能力库。

它的目标不是承载单一业务，而是把可复用的 AI 工程能力沉淀为可长期维护、可被后续项目复用的资产，包括：

- 规则
- 流程
- SOP
- Guide
- Knowledge
- Tool
- Agent 使用规范
- AI 工程方法论

它当前主要解决的问题是：

- 为 AI 提供统一的全局规则入口
- 为不同项目提供可复制的 `.ai/` 项目记忆结构
- 沉淀可复用的流程、分析资料、设计记录与操作方法
- 用工具脚本和本地工具能力支撑 AI Coding 工作

它不是什么：

- 不是 `AI Coding Console` 平台本身
- 不是项目管理系统本身
- 不是某一个具体业务项目
- 不是只服务单一代码仓库的局部配置目录

`tools/ai-coding-console` 只是这个能力库中的一个工具模块，用来承载 AI Coding 项目管理与任务工作台相关能力；它不代表整个 `ai-ui-agentic` 仓库，也不能反向定义整个仓库的顶层定位。

## 2. 核心理念

建立这个仓库的核心原因，是把 AI 工程能力从“单次对话经验”提升为“可沉淀、可追踪、可复用、可迁移”的长期资产。

这里的沉淀方式不是把所有能力都写成代码，而是按职责分层保存：

- 规则放在明确入口中，保证 AI 每次进入时都有统一边界
- 流程与 SOP 放在知识目录中，保证方法可重复使用
- 分析、审计、设计与实施记录放在追踪资料中，保证关键事实可回溯
- 通用脚本与模板放在 `tools/` 中，保证能力可以被执行而不是只停留在说明层
- 项目记忆放在 `.ai/` 中，保证 AI 进入具体项目时能快速接管上下文

这个仓库强调的不是“某一个工具跑起来”，而是“AI 工程能力能否长期复用并在不同项目之间稳定迁移”。

## 3. 核心组成

说明：以下分类是这个仓库沉淀能力时使用的概念分类；但当前仓库中并不是每一类都以同名顶层目录存在。AI 需要以“当前真实落地点”来理解，而不是把概念分类误当成现有目录结构。

### Skills

作用：
用于表达可复用的能力单元。

如何使用：
当前仓库根目录没有独立的 `skills/` 目录。现阶段与 Skill 最接近的实际载体主要有两类：

- `knowledge/flows/` 中的操作方法、流程和提示模板
- `tools/` 中可以直接执行的工具能力

另外，`tools/ai-coding-console/README.md` 中已经明确存在 Capability Registry 概念，能力类型包括 `skill`、`sop`、`script`、`prompt-template`、`capability-pack`。这说明仓库已经具备“能力分类”的设计事实，但这只是其中一个工具模块中的能力模型，不应反向等同为整个仓库已经存在独立 `skills/` 顶层目录。

### SOP

作用：
用于沉淀可重复执行的方法、步骤和边界。

如何使用：
当前 SOP/流程类内容主要落在 `knowledge/flows/`。例如当前可见的内容包括 Git 建分支提示词、市场任务处理指南，以及 `credit-card/` 下的迁移与接管类流程文档。

如果任务需要复用已有方法，应优先先读 `knowledge/flows/`，而不是直接新造流程。

### Guides

作用：
用于帮助 AI 或人快速理解某类任务、场景或工作方式。

如何使用：
当前仓库根目录没有独立的 `guides/` 目录。Guide 类内容目前分散在：

- `README.md` 这类仓库入口说明
- `knowledge/flows/` 中偏操作型的方法文档
- `knowledge/traces/` 中偏分析型、设计型、审计型的说明文档

因此，AI 在使用时应根据目的区分：

- 需要执行方法时，优先看 `knowledge/flows/`
- 需要理解背景、设计和演进脉络时，优先看 `knowledge/traces/`

### Knowledge

作用：
用于沉淀长期可复用的背景、分析、设计、迁移、审计和追踪资料。

如何使用：
当前 `knowledge/` 已分为两个明确层级：

- `knowledge/flows/`：流程、SOP、操作方法
- `knowledge/traces/`：项目、功能、仓库、工具、架构的分析与追踪资料

AI 第一次进入仓库时，如果要理解整体能力边界和历史演化，应优先查看 `knowledge/traces/` 中与个人 AI 工具库架构、能力地图、迁移结果相关的正式文档。

### Tools

作用：
用于承载可执行、可复用、可落地运行的工具能力。

如何使用：
当前 `tools/` 下已有三个实际工具目录：

- `tools/init-project-memory/`
  作用：为外部项目初始化 `.ai/` 项目记忆结构，并从模板复制基础记忆文件
- `tools/sync-codex-home/`
  作用：将仓库中的 `AGENTS.md` 和可选配置同步到本机 Codex 家目录
- `tools/ai-coding-console/`
  作用：提供 AI Coding 任务工作台相关能力，包括本地 GUI、CLI、任务能力绑定、Prompt/SOP 生成、Plan/Build Run 等

AI 应把 `tools/` 理解为“能力执行层”，而不是“业务系统目录”。

补充边界：

- `ai-coding-console` 内部可以出现 `Project`、`Task`、`Run`、`Artifact` 等概念
- 这些概念属于该工具模块的工作模型
- 它们不能被上升描述为整个 `ai-ui-agentic` 仓库的顶层定位

### Agent 使用规范

作用：
用于约束 AI 在这个仓库中的工作方式、产物要求和边界。

如何使用：
当前最核心的入口是根目录 `AGENTS.md`。它规定了：

- 复杂任务先分析后修改
- 正式结论必须落盘为仓库文档
- 目录职责必须遵守顶层结构
- `.ai/` 项目记忆是任务执行的重要上下文
- 默认以本地工具为主，避免越界扩展

因此，AI 进入本仓库时，不应把自己当作只会写代码的执行器，而应先按 `AGENTS.md` 理解仓库治理规则与产物约束。

### Project Memory

作用：
用于保存当前仓库自己的长期上下文、当前状态、决策记录和交接信息。

如何使用：
当前 `.ai/` 下已经存在：

- `business-context.md`
- `current-state.md`
- `decisions.md`
- `defect-patterns.md`
- `handoffs/`

这些文件的职责不是介绍某个业务产品，而是保存当前仓库及其工具演进过程中的稳定背景、近期状态和重要决策。AI 在开始非 trivial 任务前，应优先读取这些文件。

## 4. 当前目录职责

当前仓库里与 AI 理解最相关的目录职责如下：

- `.ai/`
  当前仓库自己的项目记忆层，保存业务背景、当前状态、决策和交接
- `knowledge/flows/`
  流程、SOP、操作方法、长期复用方法
- `knowledge/traces/`
  分析、审计、设计、迁移、实施结果与能力追踪资料
- `tools/`
  实际可执行、可复用的工具能力
- `tools/ai-coding-console/`
  AI Coding 工作台工具模块，不是整个仓库本体
- `tools/init-project-memory/`
  `.ai/` 项目记忆初始化工具
- `tools/sync-codex-home/`
  Codex 家目录同步工具
- `data/`
  持续数据目录。当前已用于 `ai-coding-console` 的项目、任务、运行记录等数据
- `config/`
  多工具共用的全局配置

根目录关键入口文件的职责如下：

- `AGENTS.md`
  全局规则入口
- `README.md`
  仓库的人类可读说明入口
- `PROJECT_MAP.md`
  当前仓库结构和关键区域的快速地图

## 5. 阅读顺序

AI 第一次进入仓库时，建议按以下顺序建立理解：

1. 先通过当前 Agent 的入口文件进入共享上下文
   典型链路：
   `AGENTS.md` / `CLAUDE.md` / 其他项目入口 → `.ai/AI_CONTEXT.md`
2. 读完本文件后，再按任务需要进入相应目录
3. 不要默认一次性加载全部目录
4. 历史 handoff 只能作为历史参考，不能自动视为当前需求
5. 如果文档之间存在冲突，先报告，再继续；不要自行改写事实

按需阅读建议：

- 需要规则边界时：读 `AGENTS.md`
- 需要整体地图时：读 `PROJECT_MAP.md`
- 需要当前状态时：读 `.ai/current-state.md`
- 需要关键决策时：读 `.ai/decisions.md`
- 需要长期背景时：读 `.ai/business-context.md`
- 需要执行流程时：读 `knowledge/flows/`
- 需要历史事实、设计、审计与实施记录时：读 `knowledge/traces/`
- 需要实际执行能力时：读 `tools/`

## 6. ai-coding-console 在整体体系中的位置

`tools/ai-coding-console` 是当前仓库中最复杂、最活跃的工具模块之一，但它只是整体能力体系中的一部分。

正确层级应理解为：

1. `ai-ui-agentic` 是个人全局 AI 工程能力库
2. `tools/` 是能力执行层
3. `tools/ai-coding-console` 是 `tools/` 里的一个具体工具
4. `data/ai-coding-console/` 是该工具对应的数据域

因此：

- 可以研究 `ai-coding-console` 的设计和实现
- 可以在 `knowledge/traces/` 中记录它的计划、实施和验证结果
- 但不能把整个仓库描述成“AI Coding Console 平台”

当前 `PROJECT_MAP.md` 和 `.ai/current-state.md` 对 `ai-coding-console` 的内容较多，反映的是当前仓库的工作重心，而不是仓库的唯一身份。

## 7. AI 第一次进入时应该怎么理解这个仓库

第一性理解原则是：

- 这里首先是能力库
- 其次才是若干具体工具的集合
- 某个工具再活跃，也不能覆盖整个仓库的定位

进入仓库后的默认心智模型应是：

1. 这是一个能力库
2. 这里保存规则、知识、方法、工具和项目记忆
3. 某些工具可能很复杂，但仍然只是能力库中的子模块
4. 任何局部实现概念，都不能自动提升为仓库总定义

## 8. 边界与非目标

AI 在这个仓库中应始终保持以下边界感：

- 不把当前仓库当成单一业务系统
- 不把 `tools/ai-coding-console` 当成仓库总定义
- 不把概念分类误写成当前真实目录结构
- 不假设当前已经存在独立 `skills/`、`sops/`、`guides/` 顶层目录
- 不把未实现的设计文档当成已交付功能
- 不把聊天结论当成正式事实来源，正式事实应以仓库中的文档产物为准

## 9. 需要确认

以下事项在当前仓库里没有看到足够强的自动化事实支撑，后续若需要进一步自动化，应先确认：

- 是否未来会恢复独立根级 `skills/`、`sops/`、`guides/` 目录
  当前事实：仓库根目录未见这些独立目录，现阶段应以 `knowledge/` 与 `tools/` 的真实落点为准
- OpenCode 是否除 `AGENTS.md` 之外还存在其他可靠的项目级自动加载入口
  当前事实：仓库内已确认 `AGENTS.md` 是现有规则入口；未发现更明确的共享项目上下文自动入口机制

## 10. 当前已确认事实与提醒

- 当前仓库的核心身份是“个人全局公用的 AI 工程能力库”
- 当前能力沉淀方式已经覆盖规则、知识、流程、工具、项目记忆和正式追踪文档
- `tools/ai-coding-console` 已经形成独立工具模块与独立数据域
- 当前仓库中不存在独立的根级 `skills/`、`sops/`、`guides/` 目录
- 当前很多近期状态文档重点围绕 `ai-coding-console`，这说明它是当前工作重点，但不应因此覆盖仓库总定位
