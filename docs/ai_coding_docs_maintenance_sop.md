# AI Coding 项目 Docs 维护 SOP

版本：v1.0  
生成日期：2026-07-08  
适用范围：使用 Claude、Codex、Cursor、Copilot、OpenHands、Devin 类 AI Coding 工具参与开发的项目

## 1. SOP 目标

这份 SOP 用于规范 AI Coding 项目的 docs 维护方式，让文档既能服务人类协作，也能服务 AI 理解项目、规划任务、生成代码、审查变更和恢复上下文。

核心原则不是“文档越多越好”，而是“关键上下文稳定、当前事实清楚、临时 Spec 可丢、长期规则要沉淀”。

## 2. 来自分享会的关键原则

AI Coding 分享会中与 docs 维护最相关的结论可以提炼为以下几点。

第一，Spec 是中间产物，不一定是长期资产。Spec 的目标是让 AI 在当前任务中知道要做什么、怎么做、边界在哪里。只要能减少返工，它就是合格的。临时 Spec 不必追求完美，也不一定要和代码长期强一致。

第二，代码和注释才是最稳定的长期上下文。AI 后续理解项目时，最常读取的是代码本身、关键注释、目录结构、测试和配置。因此 docs 不能替代代码事实，文档应尽量指向真实入口、调用链和验证方式。

第三，护栏文档比泛泛规范更重要。AI 需要知道团队偏好、禁止事项、已踩过的坑、组件复用规则、架构边界、测试要求和安全红线。这些内容应该沉淀为稳定规则，而不是每次任务都重新口头说明。

第四，复杂需求要先写状态、边界和验收。简单 CRUD 可以轻量描述，但复杂交互、复杂业务流程、数据迁移、权限、支付、回测、调度、异步任务等场景，必须先写清楚状态流、异常分支、影响范围和验收标准。

第五，AI 每犯一次可复用的错误，就要把错误模式沉淀到规则里。不要只修一次代码，而是把“以后不要再这样做”的经验写进项目规则或 SOP。

## 3. Docs 分层模型

建议每个 AI Coding 项目至少维护四类文档：入口文档、事实文档、规则文档、过程文档。

入口文档用于让新人和 AI 快速理解项目。典型文件是 `README.md`、`AGENTS.md`、`CLAUDE.md` 或 `docs/PROJECT_MAP.md`。

事实文档用于描述当前真实架构、数据流、运行方式和测试基线。典型文件是 `docs/architecture.md`、`docs/testing.md`、`docs/configuration.md`。

规则文档用于固定 AI 和人类协作时必须遵守的边界。典型文件是 `AGENTS.md`、`docs/development_rules.md`、`docs/docs_maintenance_sop.md`。

过程文档用于记录某次迁移、审计、重构、验收或事故复盘。典型文件是 `docs/task_reports/*.md`、`docs/migration_*.md`、`docs/*_result.md`。过程文档可以保留，但不能让它们取代当前事实文档。

## 4. 推荐 docs 目录结构

通用项目可以采用以下结构。

```text
README.md                         项目入口，给人类看，也给 AI 建立第一印象
AGENTS.md 或 CLAUDE.md             AI Coding 行为规则、架构边界、禁止事项

docs/
  00_project_overview.md           当前项目定位、范围、已完成能力、不包含能力
  01_architecture.md               当前真实架构、核心数据流、模块职责
  02_setup_and_run.md              安装、启动、配置、常见环境问题
  03_testing_baseline.md           默认 smoke、扩展测试、环境依赖测试
  04_development_rules.md          代码规范、提交规范、目录边界、禁止事项
  05_docs_maintenance_sop.md       docs 维护 SOP
  06_roadmap.md                    当前路线图和优先级
  task_reports/                    任务报告、验收记录、迁移记录
  archive/                         过期但暂时需要保留的历史文档
```

如果项目较小，不必一次性创建全部文件。最小可用集合是：`README.md`、`AGENTS.md`、`docs/00_project_overview.md`、`docs/01_architecture.md`、`docs/03_testing_baseline.md`。

## 5. 文档类型与维护规则

### 5.1 README.md

`README.md` 是项目总入口，必须短而准。它应该说明项目是做什么的、怎么安装、怎么启动、怎么测试、主要目录是什么、当前限制是什么。

`README.md` 不应该塞入大量历史迁移细节，也不应该记录临时任务过程。它只保留当前用户和 AI 第一时间需要的信息。

### 5.2 AGENTS.md / CLAUDE.md

这是 AI Coding 的护栏文件。它应该说明 AI 在本项目中可以做什么、不可以做什么、修改前需要怎样审计、何时必须先问用户、哪些目录职责不能跨越。

建议包括：架构边界、禁止新增的目录或抽象、删除文件前的审计要求、测试要求、输出格式、回滚要求、legacy 边界、数据目录限制。

### 5.3 架构文档

架构文档必须描述“当前真实架构”，不能只写愿景。可以保留 Future Vision，但必须明确哪些是当前已实现，哪些只是未来计划。

架构文档需要包含：核心模块、调用链、数据流、入口文件、存储位置、配置来源、外部依赖、已知边界。

### 5.4 测试基线文档

测试基线文档必须告诉 AI 和开发者：默认应该跑什么测试、哪些测试依赖真实环境、哪些测试不能默认跑。

建议分成三类：默认 smoke、独立扩展测试、环境依赖测试。

### 5.5 任务报告与验收记录

凡是涉及迁移、删除、重构、核心流程、配置、数据、状态、入口、兼容层的重要任务，都应该生成任务报告或验收记录。

验收记录至少包含：修改文件、修改摘要、验证命令、验证结果、未做事项、风险、回滚方式、是否达到验收标准。

## 6. AI Coding 任务中的 docs 使用流程

每次 AI Coding 任务建议按以下流程执行。

第一步，先读入口文档。AI 应先阅读 `README.md`、`AGENTS.md`、架构文档和测试基线，确认项目边界。

第二步，判断任务类型。如果是简单 bug fix，可以轻量处理；如果涉及多文件、迁移、删除、重构、入口、配置、数据或兼容层，必须先输出计划。

第三步，任务开始前写临时 Spec。Spec 可以放在聊天中，也可以保存到 `docs/task_reports/`。临时 Spec 的目标是讲清楚当前任务，不要求长期维护。

第四步，执行修改时优先让代码、测试和注释承载长期上下文。不要为了文档好看而让文档和代码脱节。

第五步，任务结束后决定是否沉淀文档。如果只是一次性实现，记录在任务报告即可；如果发现了可复用规则、架构边界或常见坑，则更新稳定文档。

第六步，跑验证并记录结果。验证失败不能写成“已完成”，必须记录失败原因、阻塞项和下一步。

## 7. Spec 是否要保存的判断标准

不是所有 Spec 都要保存。按以下标准判断。

应该保存的 Spec：涉及架构决策、业务规则、状态机、数据迁移、接口契约、长期维护风险、多人协作边界、验收标准。

可以不保存的 Spec：一次性小改动、局部样式调整、简单 bug fix、已经完全体现在测试和代码中的实现计划。

如果保存 Spec，建议保存为任务报告，而不是伪装成当前事实文档。例如：`docs/task_reports/2026-07-08_fix_web_smoke_report.md`。

## 8. Docs 更新触发条件

遇到以下情况必须考虑更新 docs。

项目启动方式变化时，更新 `README.md` 或 setup 文档。

依赖变化时，更新 `requirements.txt` 和安装说明。

默认测试命令变化时，更新测试基线文档。

目录职责变化时，更新架构文档和 AI 护栏文件。

新增或废弃核心入口时，更新项目地图。

发现 AI 反复犯同一类错误时，更新规则文档。

完成迁移、删除、重构或验收时，新增任务报告。

发现文档与代码不一致时，优先修正文档中的“当前事实”，不要让过期文档继续误导 AI。

## 9. Docs 不应该做的事

不要把过期计划当成当前事实。

不要把临时 Spec 强行维护成长期文档。

不要在多个文档里重复维护同一条规则。

不要让 docs 变成流水账。

不要在迁移过程中顺手改无关文档。

不要把“能跑”写成“已验收”，除非验收标准真的覆盖了业务链路。

不要把“AI 生成的说明”直接加入 docs，必须先核对代码事实。

## 10. 文档命名规范

稳定文档建议使用固定编号，便于 AI 和新人按顺序阅读。

```text
00_project_overview.md
01_architecture.md
02_setup_and_run.md
03_testing_baseline.md
04_development_rules.md
05_docs_maintenance_sop.md
06_roadmap.md
```

过程文档建议带日期、主题和结果类型。

```text
docs/task_reports/2026-07-08_requirements_readme_smoke_report.md
docs/task_reports/2026-07-08_data_service_audit.md
docs/task_reports/2026-07-08_web_smoke_validation_result.md
```

过期文档不要直接删除，除非已经完成引用审计。可以移动到 `docs/archive/`，或在文件顶部标注“历史文档，非当前事实”。

## 11. 每次任务结束后的 docs 检查清单

任务结束前，开发者或 AI 应检查以下问题。

这次是否改变了安装、启动或测试方式？如果是，更新 README 或测试文档。

这次是否改变了架构、目录职责或调用链？如果是，更新架构文档。

这次是否发现了新的 AI 规则或踩坑经验？如果是，更新 AGENTS.md 或规则文档。

这次是否涉及迁移、删除、重构、数据、配置、状态或兼容层？如果是，生成任务报告。

这次是否只是一次性实现？如果是，不要过度增加长期文档。

这次的验证结果是否清楚记录？如果失败，是否写明阻塞原因和下一步？

## 12. AI 可直接使用的 docs 维护 Prompt

可以在项目维护任务前给 AI 使用以下 Prompt。

```text
请先阅读 README.md、AGENTS.md、docs/00_project_overview.md、docs/01_architecture.md 和 docs/03_testing_baseline.md。

本次任务开始前，请判断是否需要更新 docs。判断标准：如果修改影响安装、启动、测试、架构、目录职责、配置、数据、状态、兼容层或 AI 行为规则，则必须说明要更新哪些文档；如果只是一次性局部修改，则不要新增长期文档。

执行完成后，请输出：修改文件、验证命令、验证结果、是否需要更新 docs、更新了哪些 docs、未解决问题和下一步建议。
```

## 13. 当前项目 ai-crawler 的落地建议

对 `ai-crawler`，建议将以下文档设为当前权威入口。

```text
README.md
AGENTS.md
docs/00_project_overview.md
docs/01_architecture.md
docs/TESTING_BASELINE.md
docs/ai_coding_docs_maintenance_sop.md
```

其余大量迁移、验收、冻结文档可以保留，但应逐步区分“当前事实”和“历史过程”。后续每次涉及核心流程、数据、配置、测试或兼容层的任务，都建议在 `docs/task_reports/` 下生成一份简短验收记录。

## 14. 最小执行版

如果只想记住最短版本，可以使用以下规则。

先保证 `README.md` 能让人装起来、跑起来、测起来。

再保证 `AGENTS.md` 能让 AI 知道边界、禁区和验证方式。

架构文档只写当前事实，不写空泛愿景。

Spec 只要够用，不追求永久完美。

代码、测试、关键注释才是长期上下文。

AI 犯过的通用错误，要沉淀成规则。

重要任务结束后，必须留下验收记录。
