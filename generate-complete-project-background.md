# 完整项目背景说明：整理提示词

请为我的“个人AI工具库”整理一份【完整项目背景说明】。

这份文档不是介绍个人AI工具库本身，而是让以后新开的 AI、OpenCode、不同模型或多 Session 子任务，能够快速理解我实际开发的整个微前端业务体系、开发方式和 AI 协作方式。

先阅读当前仓库已有资料、现有 SOP、现有 traces、已有工具说明和相关配置。

不要凭空补充。  
不确定的内容必须标记为“待确认”，不能自行猜测。

文档目标是让一个完全不了解背景的新 AI 读完后，能正确理解：

---

## 一、系统整体结构

说明整个新系统是怎样组成的，包括：

- workspace 的职责。
- 项目 A～G 的职责。
- 大 Menu、业务项目、小路由之间的关系。
- workspace、业务项目、市场目录之间的关系。
- 为什么一个功能可能同时涉及 workspace 和多个业务项目。
- 全量迁移项目是什么，它和项目 A～G 的关系是什么。
- 当前系统不是单体项目，而是多个独立 Git 仓库组成的微前端体系。

---

## 二、业务项目内部结构

说明每个业务项目内部：

- common 是什么。
- core 是什么。
- 各市场目录是什么。
- localchange 是什么。
- 新市场如何基于 core + common 初始化。
- 开发时如何判断应该改 core、当前市场目录还是 localchange。
- 哪些情况不能默认修改 core、workspace、其他市场或其他项目。

---

## 三、市场与本地运行方式

说明：

- 市场是开发和运行维度，不是电脑顶层目录。
- 开发某个市场功能时，通常要同时运行哪些仓库。
- 例如开发美国项目 A 时，为什么要同时运行 workspace/美国 与 项目A/美国。
- 功能联动项目 B、C 时，为什么可能需要同时启动多个项目。
- 一个 Jira 可能同时影响多个项目和多个市场目录。

---

## 四、Jira 实际开发流程

说明用户拿到一个 Jira 后，真实的定位路径和工作步骤：

```text
市场
→ 大 Menu
→ 业务项目
→ 小路由 / 具体功能
→ Jira task
```

需要说明：

- BA 通常提供什么。
- 为什么先看旧系统网站。
- 什么情况下才看旧系统代码。
- 如何判断新系统要改什么。
- 如何判断影响范围。
- 如何判断是否涉及 workspace、core、localchange、联动项目。
- 开发后如何验证。

---

## 五、旧系统网站与旧代码

说明：

- 旧系统网站是优先参考来源。
- 旧系统代码只是分析资料，不参与新系统运行。
- 一个市场可能有多个旧代码仓库。
- 一个旧代码仓库只属于一个市场。
- 为什么不能把旧代码当成新系统依赖。
- AI 查看旧逻辑时的正确优先级。

---

## 六、Playwright 的角色

说明：

- Playwright 是辅助旧系统页面和行为分析的能力。
- 用户已有独立 Playwright SOP。
- 涉及 Playwright 时必须读取并按已有 SOP 执行。
- 不重新设计字段、截图、接口、校验、行为采集规则。
- Playwright 结果用于辅助理解、迁移和验证，不替代业务判断。

---

## 七、多项目与多 Session 协同

说明真实场景：

- 一个 Jira 可能同时涉及多个业务项目。
- 也可能同时涉及旧系统分析、Playwright 采集、项目分析、代码开发、联调验证。
- 当前用户手动打开的 OpenCode 默认就是总控。
- 总控不需要额外创建。
- 总控可以创建 C0、C1 等子任务 session。
- 子任务可能来自不同项目目录。
- 用户通过 OpenCode Web 或原生 session tree 看子任务过程和结果。
- 总控不应因为子任务慢启动或慢执行被阻塞。
- 不要重复实现自定义 board 或自定义 Web 页面。

---

## 八、AI 参与开发时必须遵守的原则

明确写出：

- 先确认市场。
- 先确认 Jira / 开发目标。
- 先确认主项目和联动项目。
- 先确认对应大 Menu 和小路由。
- 先分析影响范围，再修改代码。
- 明确区分 workspace、common、core、市场目录、localchange。
- 不默认修改 workspace、core、其他市场、其他项目。
- 不删除文件、目录、配置或代码，除非明确确认。
- 旧系统逻辑优先看旧网站，必要时再看旧代码。
- 涉及 Playwright 时直接按已有 SOP。
- 多项目任务必须明确每个项目的角色和允许修改范围。
- 分析类任务优先 plan；进入 build 前必须确认修改范围。

---

## 九、个人 AI 工具库、公司 AI 工具库与 OpenCode 的关系

说明：

- 个人AI工具库是用户自己的长期开发流程、工具、知识、经验和运行记录中心。
- 公司AI工具库是团队共用的专项开发辅助工具，主要用于 localchange / 开发分析等场景。
- OpenCode 是当前使用的执行入口，不等于个人AI工具库本身。
- 个人AI工具库不是只服务公司项目，也会用于个人项目。
- 公司AI工具库不是替代个人AI工具库，而是在对应开发阶段被个人流程调用。
