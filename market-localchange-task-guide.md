# 市场 Localchange 任务处理指南

> 本文用于沉淀市场 Localchange task 的真实处理方式。  
> 它不是强制 SOP，也不是固定流水线；每个 task 仍应根据实际情况选择和调整处理方式。

---

## 1. 前置背景与适用范围

本指南基于项目整体背景使用。

处理市场 Localchange task 前，应先读取项目背景文档，理解：

- workspace、业务项目、市场、core、common 的整体关系；
- 市场目录与 core 的覆盖机制；
- 本地运行与跨项目联动方式；
- 旧系统、旧代码、Playwright 的角色；
- Jira、CB、开发环境的基本约束。

项目背景文档：

```text

```

本指南只描述：

> 在上述项目体系中，如何处理一类“市场 Localchange task”；以及 AI 在其中如何辅助分析。

## 2. 指南定位

本指南不是要求每次按固定步骤执行。

它沉淀的是处理市场 Localchange task 时，通常需要考虑的事项、范围边界和判断方式。

每个 task 的处理深度、顺序和具体方法都可能不同。

例如：

- 有时分支已经存在；
- 有时 Jira 信息较完整，有时只提供部分补充信息；
- 有时使用公司 AI 工具库分析，有时手动对比；
- 有时只改当前市场，有时需要改 core；
- 极少数 task 可能涉及多个 Journey 或多个市场，但必须按实际 task 和开发者指定处理。

---

## 3. 当前开发周期上下文

市场通常以周期方式推进。

例如，当前一段时间统一处理美国市场 task；之后切换到英国市场。

当前周期通常包含：

```text
- 当前市场
- 当前市场常用 CB
```

市场和常用 CB 不需要在每个 task 中重复输入；切换市场或周期时再更新即可。

单次 task 通常至少提供：

```text
- 当前页面开发的 Journey 名称
- 当前 Jira ticket 地址
```

Jira 中的描述、截图、评论、链接等均可作为补充信息。

---

## 4. 接收 Task

Jira 是当前 task 的入口，但 Jira 内容不是唯一的开发依据。

无论 Jira 信息多少，处理市场 Localchange task 时，仍需要结合：

```text
当前市场代码
+ core
+ 当前 Journey
+ config
+ mock
+ localchange
+ 旧系统 / 旧代码 / Playwright 等已有资料（按实际需要）
```

来判断当前差异、实际生效链路和可能修改范围。

---

## 5. 分支准备

通常在指定 CB 上处理 task。

常见处理方式：

```text
指定 CB
→ 从远程 master 创建开发分支
→ 分支名遵循团队命名规则
```

但此处不将分支创建视为每次必须机械执行的步骤。

后续可单独建设 Git 分支创建辅助能力；本指南仅保留其在 task 处理中的位置。

---

## 6. 市场 Localchange 差异判断

这是本指南的核心环节。

目标不是直接决定代码怎么改，而是先判断：

```text
当前市场相对 core，实际差异在哪里；
当前页面最终生效的是哪一层代码或配置；
本次问题可能应该通过市场覆盖解决，还是需要补 core 能力。
```

差异分析可以采用以下任一方式，按 task 实际情况选择：

```text
- 使用公司 AI 工具库进行分析；
- 手动对比市场与 core；
- 结合旧系统页面；
- 必要时查看旧系统代码；
- 必要时读取已有截图、分析资料或 Playwright 产物。
```

使用哪种方式不是强制条件；关键是完成当前市场的差异判断。

## 7. 代码与覆盖关系

### 7.1 核心结构

项目中，`core` 与各市场目录同级。

示意：

```text
项目/
├─ common/
├─ core/
│  ├─ journey/
│  ├─ config/
│  ├─ mock/
│  └─ localchange.js
├─ 美国/
│  ├─ journey/
│  ├─ config/
│  ├─ mock/
│  └─ localchange.js
└─ 英国/
   ├─ journey/
   ├─ config/
   ├─ mock/
   └─ localchange.js
```

### 7.2 各层含义

```text
core
= 标准功能、默认页面实现、默认配置、默认 mock、默认 localchange 配置

市场
= 对 core 的市场级覆盖和补充

journey
= 页面逻辑、页面组件

localchange
= 市场对 core 默认能力、开关或配置的覆盖

config / mock
= 可能影响页面行为、接口表现、展示和数据的市场或默认资料
```

### 7.3 覆盖关系

市场 Journey 存在对应代码时，可覆盖 core 的对应 Journey 代码。

市场 `localchange.js` 可覆盖 core `localchange.js` 中的同名配置。

市场 `config`、`mock` 也可能覆盖或补充 core 的默认行为。

因此，分析当前页面时，不能只看 Journey 页面代码，还需要按真实引用关系判断 `config`、`mock` 与 `localchange` 的影响。

## 8. 默认分析范围与扩展规则

### 8.1 默认范围

假设当前市场为美国，当前 task 的目标 Journey 为 `xxx`。

默认分析范围为：

```text
美国/journey/xxx
core/journey/xxx

美国/localchange.js
core/localchange.js

与 xxx 实际有关的：
- 美国/config
- core/config
- 美国/mock
- core/mock
```

不要求每次全量读取美国目录或整个 core。

应从当前目标 Journey 开始，沿真实引用、调用、跳转、配置依赖或 mock 关联逐步定位。

### 8.2 其他 Journey

默认从当前目标 Journey 开始。

当前 Journey 若真实依赖其他 Journey 的代码、组件、方法、配置或流程，则可以将这些依赖 Journey 纳入读取和分析范围。

可以扩展的依据包括：

- 当前 Journey 明确引用其他 Journey；
- 当前 Journey 明确跳转到其他 Journey；
- 当前流程必须经过其他 Journey；
- 当前配置或公共逻辑明确关联其他 Journey。

不得仅因以下原因扩展：

- Journey 名称相似；
- 功能看起来相似；
- 仅为寻找参考实现；
- AI 猜测可能有关。

是否修改依赖 Journey，由开发者最终决定。

### 8.3 其他市场

默认只处理：

```text
当前市场 + core
```

例如当前市场为美国时：

```text
可读取和分析：
- 美国
- core

不读取、不分析、不修改：
- 英国
- 其他市场
```

跨市场分析或修改必须由开发者明确指定。

AI 不得为了“找参考”或“保持一致”自行读取、分析或修改其他市场。

---

## 9. AI 在分析中的角色

AI 是分析辅助，不是最终架构决策者，也不是默认执行者。

AI 可以：

- 定位当前市场 Journey 是否覆盖 core；
- 查找当前市场与 core 的 `localchange`、`config`、`mock` 关联；
- 梳理页面、字段、按钮、接口、流程的实际生效链路；
- 识别当前 Journey 的真实依赖 Journey；
- 列出可能涉及的文件、候选修改点、影响范围和风险；
- 给出可选实现建议；
- 标记需要开发者确认的决策项。

AI 不应：

- 未经确认直接修改代码；
- 自行决定最终实现方案；
- 自行新增开关或配置；
- 自行决定默认值；
- 自行扩大到其他市场；
- 自行扩大到无关 Journey；
- 根据猜测补业务逻辑。

## 10. 开发判断

完成差异判断后，形成当前 task 的开发判断。

通常需要考虑：

```text
- 当前问题能否通过当前市场 Journey 覆盖解决；
- 是否可通过当前市场 config 解决；
- 是否可通过当前市场 localchange 覆盖解决；
- core 是否已有对应能力；
- core 是否缺少能力或配置入口；
- 是否需要在 core 补功能；
- 是否需要新增默认配置或开关；
- 当前市场是否需要覆盖开启、关闭或调整。
```

这些是开发判断项，不是自动执行规则。

最终由开发者决定：

```text
- 改哪里；
- 是否改 core；
- 是否新增开关；
- 默认值是什么；
- 当前市场覆盖什么；
- 是否涉及依赖 Journey；
- 是否扩大到跨市场范围。
```

---

## 11. Jira 补充

在完成差异判断后，视 task 实际需要更新 Jira。

可能补充：

```text
- 本次实际开发内容；
- 关键差异说明；
- 图片、截图或其他证据；
- 当前确认的范围；
- 待确认项或风险。
```

Jira 更新不是固定前置条件，也不是唯一需求来源。

它是当前 task 的记录和协作载体之一。

---

## 12. 开发与范围边界

通常可修改范围为：

```text
当前市场 + core
```

其他市场默认不修改。

当前 Journey 的依赖 Journey 是否需要修改，由开发者根据实际情况决定。

若 task 需要跨市场或跨多个 Journey 处理，应由开发者明确指定扩展范围。

---

## 13. 验证与交付

验证方式按本次实际改动决定，可能涉及：

```text
- 当前 Journey 页面逻辑与页面组件；
- 当前市场 Journey 覆盖是否生效；
- localchange / config 覆盖结果；
- mock 数据与接口表现；
- 页面字段、按钮、校验、流程；
- 当前 Journey 的依赖 Journey；
- core 默认行为是否受影响；
- 本地运行、联动项目或已有验证方式。
```

完成后，视实际需要补充 Jira、提交、MR 或其他交付信息。
